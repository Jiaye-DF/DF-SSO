Part I

商業價值與解決的問題

DF-SSO 為公司解決什麼問題、帶來什麼價值 —— 不需技術背景即可閱讀。

1專案摘要

把「每套系統各做一份登入、各記一組密碼」變成「用同一組公司帳號登入一次、到處通用、全程可稽核」的企業統一登入中心。

DF-SSO 是大豐環保的企業統一登入中心：全公司內部系統的登入一律收斂到這個中央系統，員工只用 同一組 Microsoft（Azure AD）公司帳號登入 —— 帳號開通、停用、密碼與多因素驗證（MFA）全部沿用公司既有的帳號管理，並自動帶入 ERP 的員編 / 部門等員工資料。任何新系統要提供登入，不必自己開發，向中央申請一組憑證、照整合文件接上即可。

一一次登入、到處通用

員工登入一個系統後，切換到其他已接入系統免再登入，靠中央 session 跳過 Microsoft 那一段。

集帳號集中控管

直接沿用 Microsoft 公司帳號，員工到職 / 離職由 AD 統一控管；登入紀錄、授權、限流都在中央一處。

快新系統接入快

申請一組 app_id / app_secret，照 INTEGRATION.md 接 callback / me / logout / back-channel-logout 四端點即可。

2解決的問題與商業價值

AI 輔助開發（Vibe Coding）讓公司內部系統快速增加，但每套系統的登入與權限管理各做各的、標準不一 —— 已經出現「不該看的人看到了、該看的人反而看不到」的權限失控，員工帳密散落各系統也是持續累積的資安風險。DF-SSO 把「誰能登入、看得到什麼、誰在使用」收斂回公司統一控管。

商業價值

安資安風險收斂

登入統一走 Microsoft 公司帳號 + MFA，密碼只存在一處；接入資格由中央審核，杜絕各系統自建登入的安全落差。

權權限治理有依據

所有系統基於同一份可信身分（員編 / 部門）設定權限，解決「不該看的看到、該看的看不到」—— 權限對得上組織的實際樣貌。

速系統上線更快、成本更低

新系統（含 Vibe Coding 快速產出的工具）不必重做登入，申請憑證即接即用 —— 保住開發速度，同時不犧牲安全。

稽可稽核、可問責

登入行為 100% 集中留痕，出事可追查；離職 / 停權即時全面生效，符合稽核與內控要求。

4SSO 完整 OAuth2 登入流程（§7.1.1）

完整 Authorization Code Flow，含 cookie / client_id / redirect_uri / state / client_secret / code 各驗證分支與失敗導向。每個 🛑 都是一個被擋下的攻擊面。

關鍵性質
① 中央 Redis 是登入狀態的唯一事實來源（JWT 簽名有效 ≠ 還在登入）；② 跨 App 免登入靠中央 session 跳過 Microsoft 那一段；③ 登出走中央 HMAC-SHA256 back-channel 通知所有接入 App；登出與 silent re-auth 兩種模式取捨見 docs/Design.md。

5技術亮點與工程決策

系統在安全、正確性、可維運性上的關鍵決策 —— 每一條都是「遇到什麼問題 → 怎麼解 → 帶來什麼價值」。

🛡️ 安全原語

timingSafeEqual 常數時間比對 secret / state（防 timing attack）、Redis Lua GET+DEL 原子消耗一次性 code（防重放）、back-channel HMAC-SHA256 + 30s timestamp（防偽造 / replay）、登入成功 session.regenerate（防 fixation）。

🎯 單一事實來源

登入狀態唯一事實來源是中央 Redis session，不是 JWT；白名單一張表同時驅動 OAuth 授權、Code exchange、CORS、登出廣播、redirect 驗證五件事。

⚙️ 動態 Rate Limit

四層限流不寫死，由 sso_setting 表動態載入，後台即時調整、PUT 成功後 reload 重建 limiter；DB 不可用則 fallback 內建預設值，兼顧可調與可用。

🧱 環境隔離與可維運

環境變數集中於 config/index.js 並於啟動驗證必填；雙 migration 目錄（dev 歷史 / prod baseline）；Prod / Test 完全隔離。

🔐 CSRF / Redirect 防護

Microsoft callback 的 state 以常數時間比對（防 CSRF）；redirect_uris 只存 / 比對 origin，剝除 path/query/fragment（防 open redirect），每 App 上限 10 筆、限 http/https。

🌐 前後端分離 + 動態 CORS

CORS 允許來源由白名單即時推導（快取 60 秒），新增 / 停用應用不需改 server 設定或重部署。

9核心觀念（處理需求前一定要懂的 5 件事）

  1. app_id / app_secret 是一個系統接入 SSO 的鑰匙。由後台「新增應用」時自動產生，你不用自己想。app_secret 只有在後台按「顯示」時看得到完整值，列表平常只存末四碼。
  2. redirect_uris 存的是「origin」不是完整網址。例如填 https://crm.df-recycle.com.tw，不要填 .../login/callback。一個系統最多 10 筆（給 dev/test/prod 不同網域用），只接受 http / https。
  3. 「停用」可逆，「刪除」是軟刪除。停用＝該系統暫時無法登入但資料留著；刪除＝資料庫標記，非真的清掉。兩者都不會動到使用者的 Microsoft 帳號。
  4. 登入狀態的唯一事實來源是中央的 Redis session，不是 token。所以「登出」「停用」之所以有效，是因為中央 session 被清掉 / 擋掉。
  5. 改白名單會同時影響五件事：OAuth 授權、換 token、CORS 允許來源（快取 60 秒）、登出廣播、登出 redirect 驗證。所以停用一個 App 後，最多 60 秒內它的跨網域請求才會真的被擋。

10常見使用者需求 Playbook（照著點）

Playbook A：「幫我把 ○○ 系統接進 SSO」

這是最常見的需求。你的工作是發憑證，對方工程師負責在他的系統裡接。

你在後台要做的：

  1. 進「白名單管理」分頁。
  2. 右上角點「＋ 新增應用」。
  3. 填表單：
    • 網域 *（必填）：對方系統正式網址 origin，例如 https://crm.df-recycle.com.tw。必須是合法 http/https，否則被擋。
    • 系統名稱：給人看的名字，例如「CRM 系統」。
    • 說明：一句話描述用途（可空白）。
    • Redirect URIs：每行一個 origin，dev/test/prod 各一行，最多 10 筆。例如：
      Redirect URIs
      http://localhost:3100
https://crm-test.apps.zerozero.tw
https://crm.df-recycle.com.tw
    • Frontend URL / Backend Docs URL：選填，方便日後快速點開該系統首頁 / API 文件。
  4. 按「建立應用」。系統會自動產生 app_id 和 app_secret。
  5. 在該應用那一列，點 App Secret 欄位的眼睛圖示「顯示」→ 出現完整 app_secret。
  6. 點旁邊的複製圖示，把 app_id 和 app_secret 複製下來。
  7. 安全地把這兩個值交給對方工程師，並請他讀 INTEGRATION.md 自行接入。

  • app_id、app_secret
  • SSO Backend 網址（依環境給 Prod 或 Test）
  • 一句話：「請照 INTEGRATION.md 接 callback / me / logout / back-channel-logout 四個端點」

Playbook B：「某某員工登不進去 / 某系統登入有問題」

先別急著改設定，用登入紀錄定位問題。

  1. 進「登入紀錄」分頁。
  2. 在「Email」欄輸入該員工公司 email → 按「搜尋」。
  3. 可用「狀態」「開始 / 結束日期」縮小範圍。
  4. 看最近一筆的「狀態」欄：
狀態意思通常怎麼處理
成功 successSSO 這端登入成功問題多半在對方系統那端（token 驗證 / cookie / redirect），請對方工程師查
失敗 failedMicrosoft 認證或流程失敗確認該員工 Microsoft 帳號正常、密碼沒過期；多人同時發生才可能是系統問題
ERP 未找到帳號登入了，但在 ERP 對不到員工資料該員工 ERP 員編 / 資料有問題，需找 ERP 端確認 🔲 待補：ERP 由誰維護？
  • 若完全查不到該員工任何紀錄 → 請求根本沒到 SSO，問題在對方系統「沒登入時要導來 SSO」那段沒接好，或網域沒在 Redirect URIs 裡。
  • 若整個系統所有人都登不進去 → 先確認該應用在「白名單管理」是不是被「停用」了；再看第 12 節故障排除。

Playbook C：「○○ 系統要下線 / 暫停 / 憑證外洩」

情況一：暫時停用（之後可能再開）

  1. 進「白名單管理」，找到該應用。
  2. 點該列「狀態」欄的綠色「啟用」標籤 → 變灰色「停用」。
  3. 完成。該系統最多 60 秒後無法再登入（CORS 快取到期）。要恢復就再點一次。

情況二：app_secret 外洩，要換新的

  1. 進「白名單管理」，找到該應用。
  2. 先點 App Secret 的眼睛圖示「顯示」（「重新產生」按鈕只有在顯示憑證時才出現）。
  3. 點「重新產生」→ 確認框按確定。
  4. 舊 secret 立刻失效，畫面跳出新的 secret。
  5. 立即複製新 secret，交給對方工程師更新環境變數。
⚠ 注意時機
在對方更新完成前，他的系統會換不到 token（登入會壞），所以這動作要跟對方講好時間再做。

情況三：永久下線

  1. 確認真的不要了（軟刪除可救，但別當常態操作）。
  2. 進「白名單管理」，找到該應用，點「刪除」→ 確認。

Playbook D：「新增 / 移除一位後台管理員」

「管理員」＝能登入這個後台的人，不是一般系統使用者。

新增：

  1. 進「管理員」分頁 → 右上角「新增管理員」。
  2. 只需填對方公司 email（會自動轉小寫）→ 按「新增」。
  3. 對方姓名 / Azure 資訊會在他首次登入後自動補上（在那之前狀態顯示「未登入」）。
  4. 請對方自己去管理後台用 Microsoft 帳號登入一次即可生效。

停用 / 刪除：

  • 停用（保留資料、暫時不給進）：在該管理員那列點「啟用 / 停用」標籤切換。
  • 刪除：點該列「刪除」→ 確認。
⚠ 系統禁止你刪除自己
（會跳「無法刪除自己」），這是防呆，正常現象。要移除自己請找另一位管理員操作。

Playbook E：「登入一直被擋 / too many requests / 要調次數上限」

SSO 有 Rate Limit（每個 IP 在一段時間內的請求上限），防暴力登入與 DoS。可在後台動態調整、改完立即生效。

  1. 進「設定」分頁 → 找「速率限制（Rate Limit）」區。
  2. 會看到四組可調的限制（key 與預設值見第 5 節「Rate Limit 分層」表）。
  3. 改「視窗長度（windowMs，毫秒）」或「最大請求數（max）」→ 卡片右上出現「未儲存」→ 按「儲存」。
  4. 儲存後立即生效，該限制計數會歸零重來。
⚠ 調大上限會降低安全防護
請有理由再調（例如某系統是 server-to-server 高頻呼叫才需調 session）。不確定就先別動，問前手。若資料庫暫時連不上，系統會自動 fallback 回預設值。

11維運操作（需要進伺服器 / 跑指令的）

這些不是日常點按操作，是工程師偶爾才會做的。指令細節見 CLAUDE.md。

我要做什麼怎麼做
看後端 log到 Coolify 對應 stack 看 container log
重啟服務Coolify 上 redeploy / restart 對應 stack
改環境變數Coolify 該 stack 的環境變數設定，改完要 redeploy
套用資料庫變更（migration）Prod 用 npm run migrate:up:prod；Dev/Test 用 npm run migrate:up:dev
本機跑起來開發backend/ 跑 npm run dev；frontend/ 跑 npm run dev
⚠ Migration 紅線（務必遵守）
  • 不要手動在 Prod 跑 migrate:down — Prod baseline 的 down 會 drop 掉三張表，資料全毀。
  • Schema 改動要 dev 和 prod 兩個目錄都加。
  • migration 的 down 絕不刪除管理後台自己網域的白名單資料（否則會把後台自己的存取權毀掉）。

12故障排除 Runbook

症狀可能原因先檢查什麼
所有人、所有系統都登不進去中央 Redis / Postgres 掛了，或服務沒起來Coolify 看 backend container 是否運行、Redis/Postgres 是否健康；看 backend log
單一系統全部人登不進去該 App 被停用 / 憑證被重設沒同步 / Redirect URI 沒包含其網域白名單管理確認該 App「啟用」中、Redirect URIs 含其 origin
單一使用者登不進去個人帳號 / ERP 資料問題走 Playbook B 查登入紀錄
跨系統免登入失效（每個系統都要重登）中央 session 沒建立 / cookie 被擋確認 session 正常、瀏覽器第三方 cookie 設定
出現 CORS 錯誤該系統 origin 不在白名單，或剛改完還在 60 秒快取內確認 Redirect URIs / 網域有填對，等 60 秒再試
改了 Rate Limit 沒生效—確認有按「儲存」、看到提示；理論上即時生效
登出後又自動被登回去silent re-auth 行為，屬設計取捨見 Design.md 的「登入頁兩種模式」與「Single Logout 加強模式」
大部分「登入成功但系統還是怪怪的」問題，根因在對方系統那端的接入，不在 SSO 中央。用登入紀錄確認「SSO 這端是 success」就可以把球丟回去給對方工程師查。
